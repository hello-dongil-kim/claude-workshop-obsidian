# Block 0: 왜 두 번째 뇌인가

## EXPLAIN

### 프롬프트 시대 → DB 시대

예전에는 AI를 잘 쓰는 법이 **프롬프트**였다. "너는 전문가야", "단계별로 생각해", "표로 정리해". 무슨 말을 잘하느냐가 중요했다.

그런데 Claude Code 같은 에이전트는 단순히 답만 하지 않는다. **폴더를 읽고, 파일을 고치고, 결과를 다시 남긴다.** 그래서 질문이 바뀐다.

> **"어떻게 질문할까?"보다 "무엇을 읽게 할까?"가 더 중요해진다.**

핵심 한 줄: **AI에게 말을 잘하는 게 아니라, AI가 다시 읽을 Vault(지식 창고)를 만든다.**

### RAG vs 영속 위키 — 왜 갈수록 똑똑해지나

대부분의 AI 문서 활용은 **RAG**다. 파일을 올리면 질문할 때마다 관련 조각을 찾아 답을 만든다. 매번 처음부터 다시 찾는다 — **누적이 없다.**

이 세션이 만드는 건 다르다. AI가 **영속 위키**를 점진적으로 쌓고 유지한다. 한 번 정리한 지식을 계속 갱신하므로 **쓸수록 풍부해진다.**

| 구분 | RAG (대부분의 방식) | 영속 위키 (이번 세션) |
|------|---------------------|------------------------|
| 동작 | 질문마다 원문에서 조각 재추출 | 한 번 컴파일 후 계속 갱신 |
| 누적 | 없음 (매번 처음부터) | 복리로 쌓임 |
| 예 | NotebookLM, ChatGPT 파일 업로드 | Obsidian + Claude Code |

이게 "왜 갈수록 똑똑해지나"의 답이다. frontmatter, `[[링크]]`, 정리노트가 쌓일수록 AI가 더 정확하게 맥락을 잇는다.

> 📌 **참고(개념만):** 2026년, Andrej Karpathy는 "코드보다 개인 지식DB 구축에 토큰을 더 쓴다"며 **'LLM Wiki'** 패턴을 공개했다. 이 세션은 그 흐름과 같다. 다만 우리는 영어 용어(Raw/Wiki/Schema 등)를 외우지 않고, **내 방식으로** 만든다.

### 메타포 — 작업대 / 동료 / 자산

| 요소 | 비유 | 역할 |
|------|------|------|
| **Obsidian** | 작업대 (IDE) | 내가 보고 만지는 공간 |
| **Claude** | 정리하는 동료 (프로그래머) | 요약·연결·정리를 대신 |
| **Vault** | 쌓이는 자산 (코드베이스) | 시간이 갈수록 가치가 커짐 |

### 역할 분담

- **사람이 할 일:** 소싱(자료 가져오기), 탐색, 좋은 질문하기
- **AI가 할 일:** 요약, 교차 참조, 분류, 북키핑(bookkeeping) — 사람이 귀찮아서 안 하는 잡일

이 분담이 핵심이다. 위키 관리의 가장 큰 장벽이던 "정리 비용"을 AI가 거의 0으로 낮춘다.

## QUIZ

**문제:** RAG와 "영속 위키"의 가장 큰 차이는?

| 선택지 | 내용 |
|--------|------|
| A | RAG가 항상 더 정확하다 |
| B | 영속 위키는 지식이 누적되어 갈수록 똑똑해진다 |
| C | 영속 위키는 인터넷이 필요 없다 |

**정답:** B — RAG는 질문마다 원문에서 지식을 새로 추출해 누적이 없지만, 영속 위키는 한 번 정리한 지식을 계속 갱신·축적하므로 쓸수록 풍부해진다.
