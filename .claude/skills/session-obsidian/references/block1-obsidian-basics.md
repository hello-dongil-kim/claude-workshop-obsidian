# Block 1: Obsidian 설치 + 기초

## EXPLAIN

### Obsidian = 폴더, 노트 = 마크다운 파일

Obsidian을 "예쁜 노트 앱"으로 생각하면 활용법이 좁아진다. 진짜 정체는 이렇다:

- **Vault(보관소) = 그냥 폴더** — 내 컴퓨터 안의 평범한 디렉토리
- **노트 = 마크다운(.md) 파일** — 특별한 DB가 아니라 텍스트 파일

그래서 **Claude Code 같은 에이전트가 그대로 읽고 쓸 수 있다.** 이게 "두 번째 뇌"의 기반이다. (LLM은 GitHub의 수많은 마크다운을 학습해서, 마크다운 구조를 아주 잘 이해한다.)

### 마크다운 기초 5개

| 문법 | 쓰는 법 | 결과 |
|------|---------|------|
| 제목 | `# 제목`, `## 소제목` | 계층 구조 |
| 목록 | `- 항목` | 불릿 |
| 강조 | `**굵게**` | **굵게** |
| 링크 | `[[노트이름]]` | 노트 연결(wikilink) |
| 코드 | \`\`\` 로 감싸기 | 코드 블록 |

### frontmatter — 사람도 AI도 읽는 메타데이터

노트 맨 위에 `---` 사이로 적는 YAML 블록이다. Obsidian에서는 "Properties"라고 부른다.

```yaml
---
title: 제목
type: note
tags: []
created: YYYY-MM-DD
---
```

왜 중요한가? **Claude가 본문을 다 읽기 전에 frontmatter만 보고 "이 노트가 지금 작업과 관련 있나"를 빠르게 판단**한다. 일관된 frontmatter = 빠르고 정확한 AI 검색.

### wikilink `[[ ]]` — 지식 그래프

`[[다른노트]]`로 노트를 연결하면, 흩어진 정보가 하나의 망(그래프)으로 엮인다. Claude는 이 링크를 따라가며 여러 노트에 흩어진 정보를 종합한다. 사람에겐 그래프 뷰, AI에겐 탐색 경로다.

### 사람이 직접 다루는 기본기 — 그래프·검색

Claude가 대신 찾아주기도 하지만, **내 손으로도 vault를 다닐 줄 알아야 내 것이 된다.** 세 가지면 충분하다:

> 📌 **변동 정보 (최종 확인: 2026-06-14)** — 아래 메뉴 이름·아이콘 위치·단축키는 Obsidian 버전/OS에 따라 약간 다를 수 있습니다. 안 보이면 명령 팔레트(`Cmd/Ctrl+P`)에서 "graph"·"switcher"·"search"로 검색하세요. 최신값은 [공식 도움말](https://help.obsidian.md/)을 확인하세요.

| 기능 | 단축키/위치 | 쓰임 |
|------|------------|------|
| **그래프 뷰** | 좌측 리본의 그래프 아이콘 | 노트가 점·선으로 — 지식 지도 |
| **빠른 전환** | `Cmd/Ctrl + O` | 노트 이름으로 즉시 점프 |
| **전체 검색** | `Cmd/Ctrl + Shift + F` | vault 본문 검색 |

## EXECUTE

다음을 직접 실행해보세요:

**Step 1: Obsidian 설치 + Vault 생성**
- https://obsidian.md 에서 다운로드·설치 (macOS/Windows 공통)
- 실행 → "새 보관소(Vault) 만들기" → 이름 `MyBrain`(또는 원하는 이름) → 위치 지정(예: 문서 폴더)

**Step 2: 손으로 직접 첫 노트 만들기 (도구에 손 익히기)**

먼저 내 손으로 만들어봐야 옵시디언이 손에 붙는다.
- Obsidian에서 새 노트 생성 → 파일명 `테스트노트`
- 맨 위에 frontmatter를 직접 입력:
  ```
  ---
  title: 테스트노트
  type: note
  tags: [test]
  created: (오늘 날짜)
  ---
  ```
- 본문에 `[[두번째노트]]`라고 쓰고, 그 링크를 눌러 `두번째노트`도 만들어 서로 잇기
- **그래프 뷰**를 열어 두 노트가 선으로 연결된 걸 확인 → `Cmd/Ctrl+O`로 노트 사이를 점프해보기

**Step 3: 이번엔 Claude에게 시켜보기 (같은 일을 AI가)**
> Claude Code로 vault를 다루는 방법(규칙서·안전 순서)은 **Block 3에서 정식으로 다룬다.** 여기서는 폴더만 열고 따라 해보면 된다.
- Claude Code에서 방금 만든 Vault 폴더를 연다 (File → Open Folder)
- 아래 프롬프트를 입력:
  ```
  이 vault에 "테스트노트2.md"를 만들어줘.
  맨 위에 frontmatter(title, type: note, tags, created: 오늘 날짜)를 넣고,
  본문에 [[테스트노트]] 링크를 한 줄 넣어줘.
  ```
- 손으로 만든 것과 Claude가 만든 것이 **둘 다 똑같은 .md**임을 확인 — "사람도 AI도 같은 파일을 쓴다"가 이 세션의 핵심

> ✅ **이렇게 되면 성공:** 그래프 뷰에 노트 2개가 선으로 연결돼 보이고, Claude가 만든 세 번째 노트까지 같은 vault 폴더에 `.md`로 나타나면 성공.

## QUIZ

**문제:** Obsidian이 LLM(Claude)과 궁합이 좋은 가장 큰 이유는?

| 선택지 | 내용 |
|--------|------|
| A | 그래프 뷰로 노트 관계를 시각화해줘서 |
| B | 노트를 전용 데이터베이스에 저장해 검색이 빨라서 |
| C | 노트가 마크다운(.md) 파일이라 AI가 그대로 읽고 쓸 수 있어서 |

**정답:** C — Vault는 그냥 폴더, 노트는 마크다운 파일이라 Claude 같은 에이전트가 로컬 파일을 직접 읽고 편집할 수 있다. A의 그래프 뷰는 사람용 시각화일 뿐 AI 궁합의 본질이 아니고, B는 사실과 반대다(전용 DB가 아니라 평범한 텍스트 파일이라서 좋은 것).

---

**적용 문제 (시나리오):** 친구가 "노트는 전용 DB 앱에 넣어야 검색이 빠르고 AI도 잘 읽는다"며 폐쇄형 노트 앱을 추천한다. 두 번째 뇌 관점에서 가장 적절한 판단은?

| 선택지 | 내용 |
|--------|------|
| A | 맞는 말이다. 전용 DB일수록 Claude가 더 잘 읽으니 그 앱으로 옮긴다 |
| B | 핵심은 검색 속도가 아니라 "AI가 직접 읽고 쓸 수 있는가"다. 평범한 .md 파일로 두는 편이 낫다 |
| C | 검색 속도만 빠르면 되므로, 형식은 상관없이 가장 빠른 앱을 고른다 |

**정답:** B — Claude Code가 파일을 직접 읽고 고칠 수 있어야 두 번째 뇌가 성립한다. 폐쇄형 DB는 사람 검색은 빠를 수 있어도 외부 에이전트가 그 안을 직접 다루기 어렵다. A는 "전용 DB가 AI에 유리"라는 잘못된 전제, C는 정작 가장 중요한 "AI 접근성"을 빠뜨렸다.
